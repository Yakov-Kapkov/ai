# tdd-workflow agent

A suite of five coordinated AI agents that enforce a strict Test-Driven Development lifecycle. The orchestrator drives every phase — test writing and implementation — by delegating to focused subagents. It never writes code. A companion feature-designer agent handles research and planning.

---

## Agents

| Agent | Role | Model | User-invokable | Tools |
|---|---|---|---|---|
| `tdd-workflow` | Master orchestrator | Claude Sonnet 4.6 | ✅ | read, search, agent |
| `tdd-tool-discovery` | Detects project toolchain, writes `project-tools.md` | Claude Haiku 4.5 | ❌ | read, search, edit |
| `tdd-test-writer` | Writes failing tests only (RED phase) | Claude Sonnet 4.6 | ❌ | read, edit, search, execute |
| `tdd-implementer` | Writes production code + refactors (GREEN + REFACTOR) | Claude Sonnet 4.6 | ❌ | read, edit, search, execute |
| `feature-designer` | Researches, designs, and plans features and tasks | Claude Sonnet 4.6 | ✅ | read, edit, search, web |

Only `tdd-workflow` is meant to be invoked by the user. All other agents are
internal subagents that the orchestrator calls.

---

## Workflow phases

```
PHASE 0 — BOOTSTRAP
  Detect language, verify config files, classify task type.
  If project-tools.md is missing → invoke tdd-tool-discovery.

PHASE 1 — RED
  Invoke tdd-test-writer (initial), present tests to user.
  Loop on user feedback (revision invocations) until explicit approval.

PHASE 2 — GREEN
  Invoke tdd-implementer with approved tests.
  Re-invoke if tests are still failing.

PHASE 3 — COMPLETE
  Print final summary: task, test count, coverage %, files changed.
```

### Task-type routing

| User intent | Task type | Phases run |
|---|---|---|
| Add feature / implement X | `new-feature` | 0 → 1 → 2 → 3 |
| Fix bug | `bug-fix` | 0 → 1 (repro test first) → 2 → 3 |
| Refactor (renames, extractions, signature changes) | `refactoring — code-affecting` | 0 → 1 → 2 → 3 |
| Refactor (comments, docs, formatting only) | `refactoring — cosmetic` | 0 → **skip 1 & 2** → 3 |

## Configuration

All resources are read from a `.tdd-workflow/` folder in the project root. It can be hidden(git ignored), but must be there.
The orchestrator never searches for this folder — it always uses exact paths.

| Resource | Path |
|---|---|
| Project tools (commands) | `.tdd-workflow/project-tools.md` |
| Tool-discovery spec | `.tdd-workflow/resources/{language}/tool-discovery.md` |
| Coding standards | `.tdd-workflow/resources/{language}/standards/coding-standards.md` |
| Testing standards | `.tdd-workflow/resources/{language}/standards/testing-standards.md` |
| Code style | `.tdd-workflow/resources/{language}/standards/code-style.md` |

`{language}` is inferred automatically from project markers (`package.json` →
`typescript`, `pyproject.toml` / `requirements.txt` → `python`).

### project-tools.md

Generated by `tdd-tool-discovery` on first run (after user approval). Contains
the commands the other agents use to run tests, check types, lint, measure
coverage, and run specific files. Must exist before Phase 1 proceeds.

### Standards files

Read in full by `tdd-test-writer` and `tdd-implementer`
before doing any work. Every rule in them is treated as mandatory — there are
no optional guidelines.

## Setup

To use this workflow in a project, copy the resources for your language from this
repo's [`./resources/`](../../resources/) folder into a `.tdd-workflow/` folder at
your project root. For example, for typescript the expected structure is:

```
<your-project>/
└── .tdd-workflow/
    └── resources/
        └── typescript/
            ├── tool-discovery.md
            └── standards/
                ├── coding-standards.md
                ├── testing-standards.md
                └── code-style.md
```

The copied folder contains:

| File | Purpose |
|---|---|
| `tool-discovery.md` | Spec the `tdd-tool-discovery` agent follows to detect your toolchain |
| `standards/coding-standards.md` | Mandatory coding rules for implementation and audit agents |
| `standards/testing-standards.md` | Mandatory testing rules for the test-writing agent |
| `standards/code-style.md` | Style rules (naming, comments, formatting) applied by all code agents |

## Usage

Invoke the `tdd-workflow` agent and describe your task in plain English:

```
Fix the bug where createOrder throws when quantity is 0.
```

```
Refactor UserRepository to extract a separate QueryBuilder class.
```

```
Implement the task from the current file.
```

The orchestrator will classify the task, check configuration, and drive all
subsequent phases automatically.

## Key design constraints

- **The orchestrator never writes code.** It only reads config files and
  delegates. If it detects it is about to write code, read source files, or
  run commands, it must stop and route to the correct subagent instead.
- **Each new task starts from Phase 0.** No knowledge from a previous task
  in the same chat session carries over.
- **Phases are never skipped** (except cosmetic refactoring and code-review,
  which omit phases 2 and 3 by design).
- **Tests must be failing before the implementer is invoked.** `tdd-test-writer`
  is responsible for proving the RED state before handing off.
- **The quality gate is a single-pass audit.** It does not loop. The
  implementer's own enforcement gates (tests, coverage, types, linting) run
  first; the quality gate audits what commands cannot check (magic values,
  design principles, documentation).
- **User approval is required before implementation begins.** The orchestrator
  holds at Phase 2 until the user explicitly approves the written tests.
